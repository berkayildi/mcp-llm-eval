# v0.5.0 — Retrieval and RAG evaluation

**Status:** Design spec, pre-implementation. Review before writing code.
**Scope:** Additive extension of the v0.4.x generation-eval pipeline to cover
retrieval-only metrics, full RAG end-to-end evaluation, and forward-compatible
hooks for a future Azure AI Search adapter.

---

## 1. Existing-code conventions this spec will match

After reading `judge.py`, `server.py`, `cli.py`, `types.py`, `engine.py`,
`config.py`, and the test suite, v0.5.0 code will match the following
conventions. **Deviations from the prompt are called out in bold.**

- **Dataclasses, not Pydantic.** The existing `types.py` uses stdlib
  `@dataclass` with `from_dict` / `to_dict` / `asdict` helpers. The prompt
  says "Pydantic models" but the codebase has no Pydantic dependency.
  **We will use `@dataclass` to stay consistent — introducing Pydantic for a
  handful of new types would be a gratuitous dependency and split the
  serialisation layer in two.** The new types (`RetrievedChunk`,
  `RetrievalResult`, `RAGResult`, `PoisonedChunk`) will be frozen-where-safe
  dataclasses with explicit `from_dict` / `to_dict` methods.
- **File preamble:** module docstring on line 1, then `from __future__ import
  annotations`, stdlib imports, third-party imports, local imports.
- **Module-level constants in UPPER_CASE** — e.g. `JUDGE_SYSTEM_PROMPT`,
  `JUDGE_USER_TEMPLATE`, `DEFAULT_JUDGE_MODEL`. New prompts will follow the
  same pattern.
- **Lazy SDK imports** inside `_get_*_client()` functions that raise
  `ImportError` with a `pip install <pkg>` message. `rank_bm25` will follow
  this pattern so it remains optional.
- **No `logging` module.** The code uses `print(..., file=sys.stderr)` from
  the CLI and captures errors into result fields (`EvalResult.error`) for the
  engine. New modules will do the same — no silent logger, no new dependency.
- **No retry logic** in `judge.py`. Errors propagate. Tests assert
  propagation. New judge functions will match.
- **Error handling in MCP tools:** return `types.TextContent` with an
  `"Error: ..."` string rather than raising. See `_run_evaluation` in
  `server.py`.
- **CLI subcommands:** `_cmd_<name>(args)` functions, `sys.exit(1)` for
  failure, `sys.exit(0)` for success, errors printed to `sys.stderr`.
- **MCP tool style:** `types.Tool(name=..., description=..., inputSchema={...})`
  inside `list_tools()`, dispatched in `call_tool()` with `if name == "..."`.
  JSON responses via `json.dumps(result, indent=2)`.
- **Test layout:** class-based grouping (`class TestFoo:`), `MagicMock` +
  `patch` for SDK mocks, fixture JSON under `tests/fixtures/`,
  `asyncio_mode = "auto"` (already set in `pyproject.toml`), **tests can
  import private helpers directly** — spec'd helpers are deliberately kept at
  module level.
- **Judge JSON parsing:** `raw = completion.choices[0].message.content.strip()`
  then strip markdown fences if `raw.startswith("```")` — both new judge
  functions will reuse this exact pattern (extracted to a shared helper,
  `_parse_judge_json`, kept in `judge.py`).
- **Docstrings:** short, one-line summary + optional extra paragraph. No
  multi-line reST or Google-style sections.
- **No comments explaining WHAT code does** — only WHY, and only where the
  why is non-obvious.

## 4. Design decisions (baked in)

1. **Dataclasses over Pydantic** — see §1. Justified by zero existing Pydantic
   usage; a single-source-of-truth for serialisation keeps the codebase
   simpler.
2. **Judge scoring pipeline: 1-5 integer in, 0.0-1.0 float out.**
   - Asking the judge for an integer on a 1-5 scale with anchor descriptions
     is measurably more reliable than asking for an arbitrary float. It
     reduces overconfident 0.97-style scores and forces the model to pick a
     bucket.
   - `(score - 1) / 4` maps 1→0.0, 2→0.25, 3→0.5, 4→0.75, 5→1.0.
   - **Chosen mapping: `(score - 1) / 4`, NOT `score / 5`.** A response the
     judge rates "1 — completely wrong" should score 0.0, not 0.2. The prompt
     brief says "divide by 5" which would make the worst score 0.2; I'm
     flagging this as a spec deviation and recommending `(score - 1) / 4`.
     Confirm before implementation.
   - Public surface (thresholds, reports, v0.4.x judge output) stays on
     0.0-1.0. Existing `evaluate_response` is NOT changed — it already
     returns 0-1 floats because the v0.4.x prompt asks for floats directly.
     Only the two new judge functions use the 1-5 integer protocol.
   - Rationale is documented as a docstring at the top of `judge.py`.
3. **Temperature 0 on all judge calls** — matches v0.4.x.
4. **Judge model configuration** — default `gpt-4o-mini`, overridable via
   - `judge_config["model"]` (highest precedence, matches v0.4.x behaviour),
   - `MCP_LLM_EVAL_JUDGE_MODEL` env var (new in v0.5.0, middle precedence),
   - `DEFAULT_JUDGE_MODEL` constant (`"gpt-4o-mini"`, fallback).
   A new helper `_resolve_judge_model(override: str | None) -> str` in
   `judge.py` centralises the precedence logic; both old and new judge paths
   call it.
5. **nDCG scheme: binary relevance** — simplest thing that works. Schema
   remains forward-compatible for graded relevance without breaking v0.5.0
   data: if/when graded is needed, add a parallel
   `relevant_chunk_scores: dict[str, float] | None = None` field to
   `EvalEntry`. v0.5.0 code ignores it; future code prefers it when present.
6. **Corpus format: JSONL** — one chunk per line, schema
   `{"chunk_id": str, "content": str, "metadata": dict}`. Streaming-friendly,
   diff-friendly, trivially appendable. Loader validates each line and raises
   on malformed entries.
7. **In-memory BM25** via `rank_bm25` package — zero API cost, deterministic,
   unit-testable, no network.
8. **Latency tracking** — wrap each `adapter.retrieve(...)` call in a
   `time.monotonic()` window. Collect per-query `retrieval_latency_ms` and
   compute `p50_retrieval_latency_ms`, `p95_retrieval_latency_ms` at the
   dataset level. No new tracing dependency.

## 5. `retrieval_metrics.py` — pure functions

Pure functions. No I/O, no LLM calls, no side effects. All metrics return
float in [0.0, 1.0]. Inputs: `retrieved` is ordered with rank 0 = top.

### Signatures

```python
def recall_at_k(retrieved: list[str], relevant: list[str], k: int) -> float
def precision_at_k(retrieved: list[str], relevant: list[str], k: int) -> float
def mean_reciprocal_rank(retrieved: list[str], relevant: list[str]) -> float
def ndcg_at_k(retrieved: list[str], relevant: list[str], k: int) -> float
```

### Preprocessing applied by all metrics

- `retrieved` is deduplicated **by chunk_id, keeping first occurrence**
  before any metric math. Rationale: ill-behaved retrievers sometimes return
  duplicates; they shouldn't double-count. Document this in the module
  docstring.
- `relevant` is converted to a set for O(1) membership tests.
- `k` is clamped: effective slice is `retrieved_dedup[:k]`. If
  `k > len(retrieved_dedup)`, we iterate only what exists (no padding). The
  *denominator* of precision is still `k` (standard IR convention).
- Negative or zero `k` raises `ValueError("k must be positive")`.

### Exact definitions

**Recall@k:**
```
recall@k = |retrieved[:k] ∩ relevant| / |relevant|
```
If `|relevant| == 0`: return `0.0` (caller can filter these; see §12).

**Precision@k:**
```
precision@k = |retrieved[:k] ∩ relevant| / k
```
Denominator is `k` regardless of `len(retrieved)`. If the retriever returned
fewer than `k` chunks, precision is naturally lower — this is the correct IR
semantic.

**MRR (Mean Reciprocal Rank) for a single query:**
```
mrr = 1 / (rank_of_first_relevant + 1)   if any relevant retrieved
mrr = 0                                    otherwise
```
Ranks are 0-indexed internally; the `+ 1` gives the familiar 1/1, 1/2, 1/3.
`mean_reciprocal_rank` returns this per-query value. Mean across queries is
the caller's responsibility (kept explicit so the function is single-purpose).

**nDCG@k (binary gains, log2 discount):**
```
DCG@k  = Σ_{i=0..min(k, len(retrieved))-1}  rel_i / log2(i + 2)
IDCG@k = Σ_{i=0..min(k, |relevant|)-1}       1 / log2(i + 2)
nDCG@k = DCG@k / IDCG@k                       if IDCG@k > 0
nDCG@k = 0.0                                   otherwise
```
where `rel_i = 1 if retrieved[i] in relevant_set else 0`.

The `+ 2` inside the log puts position 0's discount at `log2(2) = 1`, so the
top-rank contribution is not dampened — standard IR convention. IDCG caps at
`min(k, |relevant|)` because no retrieval can do better than placing all
relevant items at the top.

### Worked examples (ground truth for tests)

Let `retrieved = [A, B, C, D]` and `relevant = {A, C}` unless otherwise
stated.

| Case | k | Retrieved | Relevant | Recall | Precision | MRR | nDCG |
|------|---|-----------|----------|--------|-----------|-----|------|
| 1 | 2 | [A,B,C,D] | {A,C}   | 0.5    | 0.5       | 1.0 | 0.6131 |
| 2 | 4 | [A,B,C,D] | {A,C}   | 1.0    | 0.5       | 1.0 | 0.9197 |
| 3 | 3 | [B,A,C]   | {A,C}   | 1.0    | 2/3       | 0.5 | 0.6935 |
| 4 | 3 | [X,Y,Z]   | {A,B}   | 0.0    | 0.0       | 0.0 | 0.0 |
| 5 | 5 | [A]       | {A,B,C} | 1/3    | 0.2       | 1.0 | 0.4693 |
| 6 | 2 | []        | {A}     | 0.0    | 0.0       | 0.0 | 0.0 |
| 7 | 3 | [A,A,C]   | {A,C}   | 1.0    | 2/3       | 1.0 | (see below) |

Case 1 detailed nDCG math (unit-test ground truth):
```
DCG@2  = 1/log2(2) + 0/log2(3) = 1.0
IDCG@2 = 1/log2(2) + 1/log2(3) = 1 + 0.630930 = 1.630930
nDCG@2 = 1.0 / 1.630930 = 0.613147
```

Case 2 detailed nDCG math:
```
DCG@4  = 1/log2(2) + 0 + 1/log2(4) + 0 = 1.0 + 0.5 = 1.5
IDCG@4 = 1/log2(2) + 1/log2(3) + 0 + 0 = 1.630930   (only 2 relevant items)
nDCG@4 = 1.5 / 1.630930 = 0.919721
```

Case 3 detailed nDCG math:
```
DCG@3  = 0 + 1/log2(3) + 1/log2(4) = 0.630930 + 0.5 = 1.130930
IDCG@3 = 1/log2(2) + 1/log2(3) = 1.630930
nDCG@3 = 1.130930 / 1.630930 = 0.693426
```

Case 5 detailed nDCG math (k > len(retrieved)):
```
DCG@5  = 1/log2(2) = 1.0                                  (only 1 item exists)
IDCG@5 = 1/log2(2) + 1/log2(3) + 1/log2(4) = 2.130930     (3 relevant items)
nDCG@5 = 1.0 / 2.130930 = 0.469279
```

Case 7 (duplicate A at rank 1): after dedupe, `retrieved = [A, C]`, which is
case-equivalent to `[A, C]` with `relevant = {A, C}`. So `recall@3 = 1.0`,
`precision@3 = 2/3`, `MRR = 1.0`, and nDCG@3:
```
DCG@3  = 1/log2(2) + 1/log2(3) = 1.630930
IDCG@3 = 1/log2(2) + 1/log2(3) = 1.630930
nDCG@3 = 1.0
```

All seven cases will appear as unit-test fixtures with these exact expected
values (asserted with `pytest.approx(..., abs=1e-4)`).

### Edge cases explicitly tested

- `retrieved = []` → all metrics 0.0.
- `relevant = []` → recall and nDCG 0.0 (documented convention).
- `k` larger than retrieved list → handled per §5 preprocessing.
- `k = 0` or negative → `ValueError`.
- Duplicates in `retrieved` → deduped before scoring.
- `k = 1` with relevant at rank 0 → all metrics 1.0.
- No overlap between retrieved and relevant → 0.0.
- All retrieved are relevant → 1.0 for all metrics up to `k ≤ |relevant|`.
- Single-element cases for each metric (trivial but worth having).

## 6. `retrieval.py` — adapter protocol + BM25 implementation

### Protocol

```python
from typing import Protocol

class RetrievalAdapter(Protocol):
    """Synchronous retrieval interface. All adapters must implement this."""

    def retrieve(self, query: str, k: int) -> list[RetrievedChunk]:
        """Return the top-k chunks for the query, ordered by descending
        relevance score (index 0 = best match). Must return at most k chunks
        and never raise on empty-query input (returns [])."""
```

Synchronous deliberately — v0.5.0 runs single-threaded, and the BM25 adapter
has no I/O. Future async adapters (e.g. Azure AI Search) can implement a
thin sync wrapper or we can add `AsyncRetrievalAdapter` in a later release.

### `RetrievedChunk` dataclass (defined in `types.py`, re-exported here)

```python
@dataclass
class RetrievedChunk:
    chunk_id: str
    content: str
    score: float
    metadata: dict[str, Any] = field(default_factory=dict)

    def to_dict(self) -> dict[str, Any]: ...
    @classmethod
    def from_dict(cls, data: dict[str, Any]) -> RetrievedChunk: ...
```

### `BM25Adapter`

```python
class BM25Adapter:
    def __init__(
        self,
        corpus: list[dict[str, Any]] | None = None,
        corpus_path: str | None = None,
    ): ...

    def retrieve(self, query: str, k: int) -> list[RetrievedChunk]: ...

    @classmethod
    def from_jsonl(cls, path: str) -> BM25Adapter: ...
```

Construction accepts either an in-memory list of chunk dicts OR a JSONL path.
Exactly one must be provided; providing both or neither raises `ValueError`.
The JSONL loader validates each line, raises `ValueError(f"Line {n}: ...")`
on malformed entries (missing `chunk_id`, missing `content`, non-dict
`metadata`).

**SDK import pattern** (matches `judge._get_openai_client`):

```python
def _get_bm25():
    try:
        from rank_bm25 import BM25Okapi
    except ImportError as e:
        raise ImportError(
            "rank_bm25 not installed. In-memory retrieval requires it: "
            "pip install rank_bm25"
        ) from e
    return BM25Okapi
```

### Tokenisation

Deliberately simple to avoid NLTK / spaCy dependencies:

```python
def _tokenize(text: str) -> list[str]:
    # lowercase, strip ascii punctuation (keep apostrophes-in-words),
    # split on whitespace
    return [t for t in re.split(r"[^\w']+", text.lower()) if t]
```

Documented in the module docstring: "Tokenisation is intentionally naive.
For production use, plug in a domain-specific tokeniser by subclassing
`BM25Adapter` and overriding `_tokenize`."

### Tie-breaking and determinism

`rank_bm25` returns a float score per document. After scoring:

```python
indexed = [(score, i, chunk_id) for i, (score, chunk_id) in enumerate(...)]
indexed.sort(key=lambda t: (-t[0], t[1]))   # descending score, ascending orig idx
```

Secondary sort on original corpus index ensures deterministic, reproducible
ordering regardless of Python dict ordering or BM25 internals.

### Edge cases

- Empty query (`""` or whitespace-only) → returns `[]`. Does not call BM25.
  This is documented explicitly because some adapters would raise; we want
  graceful degradation.
- Query with no tokens in common with any chunk → BM25 returns all-zero
  scores. We still return the top-k *by index* with score 0.0, so callers
  can tell the difference between "didn't retrieve" and "retrieved but
  unrelated".
  - **Decision to confirm:** alternative is to filter out zero-score chunks
    and return `[]`. I'm picking "return top-k with score 0" because it
    preserves the k-slot contract and mirrors what Azure AI Search does.
    Flag if you'd prefer the filter behaviour.
- Empty corpus → adapter construction raises `ValueError("corpus is empty")`.
- `k > len(corpus)` → returns all chunks (order is still deterministic).
- `k <= 0` → `ValueError("k must be positive")`.
- Duplicate `chunk_id` in corpus → construction raises `ValueError(f"Duplicate
  chunk_id: {id}")`. Ambiguous IDs would corrupt retrieval metrics.

### Future `AzureSearchAdapter` hook

No code in v0.5.0, but the Protocol is designed to accept one without
schema changes. Expected shape for Phase 2:

```python
class AzureSearchAdapter:
    def __init__(self, endpoint: str, index: str, api_key: str): ...
    def retrieve(self, query: str, k: int) -> list[RetrievedChunk]: ...
```

Same `retrieve()` signature, same `RetrievedChunk` return shape. This is why
the protocol method is sync — Azure's SDK has a sync client.

## 7. Judge extensions in `judge.py`

Two new public functions. Each has a module-level prompt constant, takes an
OpenAI client as its first argument (matching `evaluate_response`), and
returns a float in [0.0, 1.0].

### Shared helper (new)

Extract the existing markdown-fence stripping + JSON-parse logic from
`evaluate_response` into a private helper, then have all three judge
functions use it:

```python
def _parse_judge_json(raw: str) -> dict[str, Any]:
    """Strip markdown fences from raw judge output and parse as JSON.
    Raises json.JSONDecodeError on malformed output."""
    raw = raw.strip()
    if raw.startswith("```"):
        raw = raw.split("\n", 1)[1] if "\n" in raw else raw[3:]
        if raw.endswith("```"):
            raw = raw[:-3]
        raw = raw.strip()
    return json.loads(raw)
```

The existing `evaluate_response` is refactored to use this helper. Behaviour
is unchanged; tests pass unmodified. This is a zero-risk refactor that
keeps the new judge code DRY.

### Mapping helper (new)

```python
def _scale_1_to_5(score: int | float) -> float:
    """Map a judge's 1-5 integer score to a float in [0.0, 1.0].
    Formula: (score - 1) / 4. Raises ValueError if score is out of range."""
    s = float(score)
    if s < 1.0 or s > 5.0:
        raise ValueError(f"Judge score out of range [1, 5]: {score}")
    return round((s - 1.0) / 4.0, 4)
```

### `judge_context_relevance`

```python
CONTEXT_RELEVANCE_SYSTEM_PROMPT = """\
You are an expert evaluator scoring retrieval quality for a
question-answering system.

You will receive a user query and a single retrieved text chunk. Score how
relevant the chunk is to the query on an integer scale from 1 to 5:

**5 — Highly relevant.** The chunk directly answers the query or contains
the key information needed to answer it.
**4 — Relevant.** The chunk is clearly on-topic and contributes useful
supporting information, even if it is not the complete answer.
**3 — Marginally relevant.** The chunk touches on the topic but would not,
on its own, be enough to answer the query.
**2 — Weakly relevant.** The chunk shares surface-level keywords with the
query but does not contain information that would help answer it.
**1 — Not relevant.** The chunk is off-topic, unrelated, or misleading.

Return ONLY a JSON object with exactly these keys:
{"score": <integer 1-5>, "reason": "<one-sentence justification>"}"""

CONTEXT_RELEVANCE_USER_TEMPLATE = """\
## Query
{query}

## Retrieved Chunk
{chunk}

Score the chunk's relevance to the query. Return only the JSON object."""


def judge_context_relevance(
    client: Any,
    query: str,
    chunk: str,
    judge_model: str | None = None,
    temperature: float = 0,
) -> dict[str, Any]:
    """Return {"score": float 0-1, "reason": str, "raw_score": int 1-5,
    "judge_model": str}. See module docstring for rationale."""
```

- **Model:** resolved via `_resolve_judge_model(judge_model)`.
- **Temperature:** 0 (hard default; callable override exists only for
  parity with `evaluate_response`).
- **Max tokens:** 150 (single-sentence reason + one integer — half of
  `evaluate_response`'s 300).
- **Parsing:** `_parse_judge_json(raw)`, then `_scale_1_to_5(data["score"])`.
- **Malformed JSON:** `json.JSONDecodeError` propagates (matches existing
  behaviour; caller decides how to handle).
- **Out-of-range score:** `ValueError` propagates from `_scale_1_to_5`.
- **Missing keys:** `KeyError` propagates.
- **Refusal:** OpenAI `gpt-4o-mini` doesn't refuse scoring queries in
  practice. If it ever does, the output won't parse as our expected schema
  and `json.JSONDecodeError` or `KeyError` surfaces — we document this as
  the "refusal path" rather than adding custom refusal-detection code.

### `judge_citation_faithfulness`

```python
CITATION_FAITHFULNESS_SYSTEM_PROMPT = """\
You are an expert evaluator scoring factual faithfulness of answers
produced by a retrieval-augmented generation (RAG) system.

You will receive the generated answer and the list of chunks that were
cited as its sources. Score how faithfully the answer is supported by the
cited chunks on an integer scale from 1 to 5:

**5 — Fully faithful.** Every factual claim in the answer is directly
supported by the cited chunks. No fabricated details.
**4 — Mostly faithful.** Nearly all claims are supported; any unsupported
content is minor (e.g., a connective phrase, a rewording) and does not
introduce new facts.
**3 — Partially faithful.** Some claims are supported by the chunks and
some are not. The unsupported claims are not clearly contradicted by the
chunks but are not present in them either.
**2 — Weakly faithful.** A substantial portion of the answer is
unsupported or extrapolated well beyond what the chunks state.
**1 — Unfaithful.** The answer contains fabricated facts, contradicts the
cited chunks, or is essentially unsupported.

Return ONLY a JSON object with exactly these keys:
{"score": <integer 1-5>, "reason": "<one-sentence justification>"}"""

CITATION_FAITHFULNESS_USER_TEMPLATE = """\
## Generated Answer
{answer}

## Cited Chunks
{chunks}

Score how faithfully the answer is supported by the cited chunks. Return
only the JSON object."""


def judge_citation_faithfulness(
    client: Any,
    answer: str,
    cited_chunks: list[str],
    judge_model: str | None = None,
    temperature: float = 0,
) -> dict[str, Any]:
    """Return {"score": float 0-1, "reason": str, "raw_score": int 1-5,
    "judge_model": str}."""
```

- `cited_chunks` is rendered into the user message as
  `"\n\n---\n\n".join(f"[chunk {i+1}]\n{c}" for i, c in enumerate(chunks))`.
- Empty `cited_chunks` → skip the LLM call, return
  `{"score": 0.0, "reason": "No chunks cited", "raw_score": 1,
  "judge_model": <resolved>}` directly. This avoids burning a judge call
  on an obviously-unfaithful case.
- Same model/temperature/parsing rules as `judge_context_relevance`.
- **Max tokens:** 200 (answer + chunks are longer; judge may need slightly
  more room to justify).

### Docstring at top of `judge.py`

A new module-level docstring block documents:

1. The float 0-1 convention for all public APIs.
2. Why the new judges use integer 1-5 internally (measurement reliability,
   calibration against human ratings).
3. The `(score - 1) / 4` mapping and why we picked it over `score / 5`.
4. Temperature 0 on all judge calls is a hard requirement.
5. The `_resolve_judge_model` precedence chain.

## 8. Types (`types.py` extensions)

All new types are `@dataclass`, with `from_dict` / `to_dict` helpers matching
existing style. All fields are JSON-serialisable.

### New: `RetrievedChunk`

```python
@dataclass
class RetrievedChunk:
    chunk_id: str
    content: str
    score: float
    metadata: dict[str, Any] = field(default_factory=dict)
```

### New: `RetrievalResult`

One query's retrieval pass.

```python
@dataclass
class RetrievalResult:
    query_id: str                       # EvalEntry.id
    query: str
    retrieved_chunk_ids: list[str]      # ordered, rank 0 = top
    retrieval_latency_ms: float
    metrics: dict[str, float] = field(default_factory=dict)
    # metrics keys: "recall_at_k", "precision_at_k", "mrr", "ndcg_at_k"
    k: int = 0
    error: str | None = None
```

`retrieved_chunk_ids` is the minimal shape needed to recompute any metric
later; full `RetrievedChunk` objects are stored in `RAGResult` only, to keep
pure-retrieval reports small.

### New: `RAGResult`

Full end-to-end pipeline result for one query.

```python
@dataclass
class RAGResult:
    query_id: str
    query: str
    # retrieval
    retrieved_chunks: list[RetrievedChunk]
    retrieval_latency_ms: float
    retrieval_metrics: dict[str, float]     # recall@k, precision@k, mrr, ndcg@k
    # generation
    model: str
    provider: str
    answer: str | None = None
    generation_metrics: dict[str, Any] = field(default_factory=dict)
    # keys: input_tokens, output_tokens, time_to_first_token_ms,
    #       total_latency_ms, cost_per_query, stop_reason
    # judge
    context_relevance_score: float | None = None     # 0-1
    context_relevance_reason: str | None = None
    citation_faithfulness_score: float | None = None # 0-1
    citation_faithfulness_reason: str | None = None
    judge_model: str | None = None
    # bookkeeping
    error: str | None = None
```

`context_relevance_score` is computed as the **mean** of per-chunk
`judge_context_relevance` scores, so the judge runs once per retrieved
chunk per query. Rationale: if we average first, a single irrelevant chunk
dilutes an otherwise-good retrieval; explicit per-chunk scores let us see
which chunks dragged the score down. The mean lives in the aggregate;
per-chunk scores are kept in `generation_metrics["per_chunk_context_scores"]`
as a list of floats for later inspection.

### New: `PoisonedChunk` (stub only, no logic in v0.5.0)

```python
@dataclass
class PoisonedChunk:
    """Stub for Phase 2 corpus poisoning. Fields reserved so v0.5.0 datasets
    that declare poisoned_chunks deserialise cleanly."""

    chunk_id: str
    poison_type: str                # e.g. "contradiction", "injection"
    payload: str                    # what was injected
    expected_detection: bool = True # should the pipeline flag it?
```

### Extended: `EvalEntry`

```python
@dataclass
class EvalEntry:
    id: str
    category: str
    context: str
    question: str
    expected_response: str
    tags: list[str] = field(default_factory=list)
    # NEW in v0.5.0 — all optional, None by default
    relevant_chunk_ids: list[str] | None = None
    poisoned_chunks: list[PoisonedChunk] | None = None
```

`from_dict` updates:
```python
relevant_chunk_ids=data.get("relevant_chunk_ids"),
poisoned_chunks=[PoisonedChunk(**p) for p in data["poisoned_chunks"]]
    if data.get("poisoned_chunks") else None,
```

**Backwards compat:** v0.4.x datasets never set these fields, so `.get()`
returns `None` and the entry behaves exactly as before. Engine code paths
that want retrieval eval check `if entry.relevant_chunk_ids is not None`.

### Forward-compat hook for graded relevance

Not implemented in v0.5.0, but the shape is reserved:

```python
# relevant_chunk_scores: dict[str, float] | None = None  # FUTURE v0.6.x
```

Documented in `types.py` as a comment only, so future readers understand the
extension point. No code paths reference it in v0.5.0.

**Purpose:** Full pipeline — retrieve, generate, judge. One tool call runs
all three phases for a dataset.

```python
types.Tool(

        "type": "object",

            "corpus_path":  {"type": "string"},

            "k":            {"type": "integer", "default": 5},
            "adapter":      {"type": "string", "enum": ["bm25"], "default": "bm25"},

## 11. Eval gate extension (`config.py`, `engine.check_thresholds`)

### New threshold keys

Add to `.eval-gate.yml` `thresholds:` block (all optional):

- `avg_recall_at_k`               — minimum, fail if `actual < threshold`
- `avg_precision_at_k`            — minimum, fail if `actual < threshold`
- `avg_mrr`                       — minimum, fail if `actual < threshold`
- `avg_ndcg_at_k`                 — minimum, fail if `actual < threshold`
- `avg_context_relevance`         — minimum, fail if `actual < threshold`
- `avg_citation_faithfulness`     — minimum, fail if `actual < threshold`
- `p95_retrieval_latency_ms`      — maximum, fail if `actual > threshold`

### `ThresholdConfig` extension

```python
@dataclass
class ThresholdConfig:
    # existing (v0.4.x)
    avg_faithfulness: float | None = None
    avg_relevance: float | None = None
    p95_ttft_ms: int | None = None
    max_cost_per_query: float | None = None
    # new (v0.5.0)
    avg_recall_at_k: float | None = None
    avg_precision_at_k: float | None = None
    avg_mrr: float | None = None
    avg_ndcg_at_k: float | None = None
    avg_context_relevance: float | None = None
    avg_citation_faithfulness: float | None = None
    p95_retrieval_latency_ms: int | None = None
```

`from_dict` adds `.get(...)` for each new key. Existing v0.4.x configs parse
unchanged (new fields stay `None` → check skipped).

### `check_thresholds` extension

Same pattern as existing: for each non-None threshold, compute `actual`,
compare, append a `MetricCheck`. Retrieval / RAG metrics are read from the
`overall` block of a RAG summary (or a retrieval summary for retrieval-only
checks). `check_thresholds` auto-detects summary type by checking for the
presence of `avg_recall_at_k` keys in the overall block.

### `.eval-gate.yml` example combining v0.4.x and v0.5.0

```yaml
dataset: eval/rag_dataset.json
corpus:  eval/corpus.jsonl        # NEW — v0.5.0 config key

output_dir: eval/results

models:
  - provider: anthropic
    model: claude-sonnet-4-6
    input_cost_per_mtok: 3.0
    output_cost_per_mtok: 15.0
  - provider: openai
    model: gpt-4o-mini
    input_cost_per_mtok: 0.15
    output_cost_per_mtok: 0.60

judge:
  provider: openai
  model: gpt-4o-mini
  temperature: 0

retrieval:                         # NEW — v0.5.0 config block
  adapter: bm25
  k: 5

thresholds:
  # v0.4.x — generation
  avg_faithfulness: 0.80
  avg_relevance: 0.75
  p95_ttft_ms: 500
  max_cost_per_query: 0.01
  # v0.5.0 — retrieval
  avg_recall_at_k: 0.75
  avg_precision_at_k: 0.50
  avg_mrr: 0.70
  avg_ndcg_at_k: 0.75
  p95_retrieval_latency_ms: 50
  # v0.5.0 — RAG
  avg_context_relevance: 0.70
  avg_citation_faithfulness: 0.80
```

### `config.py` parser changes

- Accept new optional top-level keys: `corpus`, `retrieval`.
- `retrieval` block defaults: `{"adapter": "bm25", "k": 5}`.
- All new threshold keys are optional; parser inserts `None` defaults into
  the returned dict.
- No new *required* fields — v0.4.x configs continue to load unchanged.

## 12. Test strategy

### Target test count

Existing baseline: 260 tests. Target: **40–50 new**, totalling 300–310.

### Unit tests

**`tests/test_retrieval_metrics.py`** (≈15 tests):

Each metric gets a dedicated `TestMetric` class. Cases 1–7 from §5 appear
verbatim. Additional per-metric edge cases:
- `recall_at_k`: empty relevant → 0.0; all-hit at k=1; k larger than
  retrieved; duplicate retrieved dedupe.
- `precision_at_k`: k > retrieved → denominator still k; empty retrieved;
  k=0 raises `ValueError`.
- `mean_reciprocal_rank`: first at rank 0 → 1.0; first at rank 3 → 0.25;
  no match → 0.0; relevant empty → 0.0.
- `ndcg_at_k`: perfect ranking → 1.0; reverse ranking gives expected
  sub-1.0 value computed from formula; IDCG=0 → 0.0; duplicate dedupe.

**`tests/test_retrieval.py`** (≈10 tests):

- `TestBM25Adapter`:
  - construction from in-memory corpus
  - construction from JSONL file
  - both-or-neither raises
  - empty corpus raises
  - duplicate `chunk_id` in corpus raises
  - malformed JSONL line raises with line number
  - `retrieve` returns ≤ k chunks
  - `retrieve` returns ordered by score descending
  - empty query → `[]`
  - unrelated query → top-k with score 0
  - `k <= 0` raises
  - deterministic ordering (same corpus + query → identical rank order across runs)

**`tests/test_judge.py` extensions** (≈10 tests):

- `TestParseJudgeJson`: fences, no fences, malformed raises, empty raises.
- `TestScale1To5`: 1→0.0, 3→0.5, 5→1.0, 0 raises, 6 raises, float 2.5 → 0.375
  (we accept numeric judges that sometimes return floats).

  malformed JSON; uses `gpt-4o-mini` by default; override via arg; override

## 14. Rollout plan and ordering

Ship order (each step merges independently; each passes CI on its own):

1. **Types + dataset schema** (§8) — foundation for everything else.
   - `EvalEntry` gains optional fields.
   - New dataclasses added with `from_dict` / `to_dict`.
   - Round-trip tests land here.
2. **Retrieval metrics module** (§5) — pure functions, no dependencies
   beyond stdlib `math`. Parallel-safe to land next to types.
3. **Retrieval adapter + BM25** (§6) — adds `rank_bm25` dependency. Lands
   after metrics so BM25 tests can assert metric outputs.
4. **Judge extensions** (§7) — independent of retrieval, but the RAG
   pipeline in step 5 depends on it. Lands before pipeline.
5. **Engine + MCP tools + CLI** (§9, §10) — wires the previous four steps.
   `engine.run_rag_evaluation` function + 4 new MCP tools + 2 new CLI
   subcommands land together because they share the orchestration path.
6. **Eval gate parser extension** (§11) — threshold keys, config block,
   `.eval-gate.yml` example. Could land in step 5 but split for reviewability.
7. **Fixtures + integration tests** (§12) — corpus, dataset, canned
   adapter, end-to-end tests. Lands after engine so canned adapter can
   plug into real code paths.
8. **Docs + release** — README section, CHANGELOG entry, version bump to
   `0.5.0`, PyPI release via existing release-please workflow.

### Ordering constraints

- **Step 1 MUST land first.** All other modules import from `types.py`.
- **Step 2 before step 3:** BM25 adapter tests assert metric values.
- **Step 4 before step 5:** RAG pipeline uses judge extensions.
- **Step 5, 6 interchangeable:** no hard dep, but step 5 PR is cleaner if
  step 6 is already in so `.eval-gate.yml` threshold fields exist for
  example configs.
- **Step 7 after step 5 and 6:** integration tests touch both.
- **Step 8 last.** Don't bump version until everything is green.

### Risk hot-spots

- **Judge scale decision (§4, point 2):** confirm the
  `(score - 1) / 4` mapping before implementing step 4. The prompt says
  "divide by 5" but that yields 0.2 for the worst score, which contradicts
  the "float 0-1" framing. Cheap to change later but better to get right
  first time.
- **Context rendering into system prompt (§9.2):** the exact template is
  user-visible (it ends up in the context window). Worth reviewing during
  step 5 PR.
- **BM25 zero-score behaviour (§6):** flagged for confirmation. If you
  prefer "filter out zero-score", change the sort/slice in one place.
